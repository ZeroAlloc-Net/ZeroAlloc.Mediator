# ZeroAlloc.Mediator 3.1.0 — DI Integration Design

**Issue:** [#59 — DI integration: auto-register handlers and eliminate parameterless constructor requirement](https://github.com/ZeroAlloc-Net/ZeroAlloc.Mediator/issues/59)

**Status:** Design — pending implementation

## Goal

Make `IMediator`-injected ASP.NET / hosted-app usage first-class, eliminate per-handler boilerplate, and remove the unsafe `null!` parameterless-constructor pattern — without compromising the zero-allocation static `Mediator.Send()` path.

## Core Principle: Two Parallel Paths

3.1.0 formalises a split that already implicitly existed:

| Path | Use case | Hot-path cost |
|---|---|---|
| **Static** — `Mediator.Send(req, ct)` | Libraries, perf-critical code, non-DI hosts, zero-alloc workloads | ~2 ns, no allocation |
| **DI** — `IMediator.Send(req, ct)` injected | ASP.NET Core, generic host, anything with scoped services | One DI resolution per call (matches MS DI norms) |

Today the DI path delegates to the static path, defeating the point. After 3.1.0, the two are parallel implementations generated side-by-side. Users pick per call site.

## Architectural Changes

### 1. `MediatorService` resolves from its own scope

**Before:**
```csharp
// Singleton; calls static dispatcher
public class MediatorService : IMediator
{
    public ValueTask<TResponse> Send<TRequest>(TRequest req, CancellationToken ct)
        => Mediator.Send(req, ct);   // → static factory fields
}
```

**After:**
```csharp
// Scoped; resolves from injected provider (= request scope in ASP.NET)
public partial class MediatorService : IMediator
{
    private readonly IServiceProvider _services;
    public MediatorService(IServiceProvider services) => _services = services;

    public ValueTask<TResponse> Send(MyRequest req, CancellationToken ct)
    {
        var handler = _services.GetRequiredService<MyRequestHandler>();
        // pipeline behaviors also resolved from _services
        return handler.Handle(req, ct);
    }
    // … one typed Send/Publish/CreateStream per discovered handler, generated
}
```

`AddMediator()` registration changes:
- `IMediator → MediatorService` was `Singleton`; becomes `Transient`. The mediator does not own a scope — it consumes whatever scope the caller is in (ASP.NET request scope, an `IServiceScopeFactory.CreateScope()` from a worker, etc.). Transient inherits the caller's `IServiceProvider`, so handlers resolve from the same scope as the consumer. Matches MediatR conventions; strictly safer than Scoped (no captive-dependency footgun if a Singleton injects `IMediator`).

### 2. Assembly-scan handler registration

```csharp
services.AddMediator()
    .RegisterHandlersFromAssembly(typeof(Program).Assembly);

// or
services.AddMediator()
    .RegisterHandlersFromAssemblies(typeof(Program).Assembly, typeof(Module).Assembly);

// override default lifetime (Transient)
services.AddMediator()
    .RegisterHandlersFromAssembly(asm, ServiceLifetime.Scoped);
```

Implementation: extension methods on `IMediatorBuilder`, hand-written (not generated) — one-time reflection at startup over `IRequestHandler<,>` / `INotificationHandler<>` / `IStreamRequestHandler<,>`. Acceptable because it runs once at startup, not on the hot path.

`[HandlerLifetime(ServiceLifetime)]` attribute on a handler class wins over the registration default:

```csharp
[HandlerLifetime(ServiceLifetime.Scoped)]
public class GetOrderHandler : IRequestHandler<GetOrderQuery, Order> { … }
```

### 3. Conditional `new T()` fallback (issue ask 2)

Today, generator unconditionally emits:
```csharp
var handler = _myHandlerFactory?.Invoke() ?? new MyHandler();
```

After 3.1.0:

- If `MyHandler` has an accessible parameterless constructor → emit unchanged. No breaking change for handlers that genuinely have one.
- If `MyHandler` has only parameterised constructors → emit:
  ```csharp
  var handler = _myHandlerFactory?.Invoke()
      ?? throw new InvalidOperationException(
          "No factory registered for MyHandler. " +
          "Either inject IMediator (recommended for ASP.NET / hosted apps), " +
          "call services.AddMediator().RegisterHandlersFromAssembly(...), " +
          "or call Mediator.Configure(c => c.SetFactory<MyHandler>(() => new MyHandler(...))).");
  ```

Users with constructor-injected handlers no longer need the unsafe `null!` shim ctor. They register via DI and inject `IMediator`, or call `SetFactory` for the static path.

### 4. Compile-time guidance: analyzer `ZAM007`

New diagnostic in `ZeroAlloc.Mediator.Generator`:

```
ZAM007: Handler 'MyHandler' has no parameterless constructor.
        The static Mediator.Send() will throw at runtime unless a factory is set
        via Mediator.Configure(...) or services.AddMediator().RegisterHandlersFromAssembly(...).
        Severity: Warning.
```

Fires when a handler has only parameterised constructors. Catches the issue at build time. Suppressible per-handler with `#pragma warning disable ZAM007` for users who know they always go through DI.

### 5. Static-API path stays untouched

No `app.UseMediator()`, no `Mediator.SetServiceProvider(...)`, no AsyncLocal scope. Strict separation. The static API keeps its current contract: zero allocation, no DI awareness, no scoping. Documentation calls this out clearly:

> **Static `Mediator.Send()`** — for libraries, workers without scoped services, perf-critical code paths. Wire factories via `Mediator.Configure(...)` or accept the parameterless-ctor fallback.
>
> **Injected `IMediator.Send()`** — for ASP.NET Core, generic host, anything with scoped services. Wire via `services.AddMediator().RegisterHandlersFromAssembly(...)`. Handlers resolved from current scope.

## Generator Changes Summary

`MediatorGenerator` will emit additional output:

1. Per-handler typed methods on `MediatorService` that resolve from `_services` (scope) instead of delegating to the static dispatcher. Includes pipeline behavior support, parallel/sequential notifications, streams.
2. Conditional `new T()` fallback (drop when no parameterless ctor accessible).
3. ZAM007 diagnostic when no parameterless ctor + no obvious DI registration path detected.

`IMediatorBuilder` extension class (hand-written, not generated):
- `RegisterHandlersFromAssembly(Assembly, ServiceLifetime = Transient)`
- `RegisterHandlersFromAssemblies(params Assembly[])`

`[HandlerLifetime]` attribute (hand-written, in `ZeroAlloc.Mediator` namespace).

## Breaking changes (3.0.x → 3.1.0)

1. **`MediatorService` registered as Transient, not Singleton.** Anyone who captured the singleton instance in a long-lived field will get a fresh instance per injection. Behaviorally equivalent (handler dispatch is stateless), but reference-equality checks would break.
2. **No more `?? new T()` fallback for handlers without parameterless constructors.** Static `Mediator.Send()` throws at runtime where it would previously have failed at compile time (handlers with no parameterless ctor never compiled before). Net: _fewer_ compile errors, _more_ runtime errors when misconfigured. Migration: register via `RegisterHandlersFromAssembly` or call `SetFactory`.

## Non-goals

- AsyncLocal-based scope wiring for static API.
- ASP.NET-specific bridge package.
- Auto-registration of pipeline behaviors via assembly scan (separate ticket; today they're discovered by attribute).
- Reflection-free assembly scan. Assembly scan is startup-only — reflection cost is acceptable. The hot path stays zero-alloc.
- Generic `Send<TResponse>(IRequest<TResponse>)` runtime dispatch on `IMediator`. Today's typed-per-handler API stays.

## Benchmarks

Existing `tests/ZeroAlloc.Mediator.Benchmarks/Program.cs` compares static `Mediator.Send` vs MediatR and has a single `SendDI` category. After 3.1.0 the DI path is no longer a thin wrapper around the static dispatcher, so we need fresh measurements.

**Add categories** (each with `[Baseline]` static, `_DI` ZeroAlloc, MediatR comparison):

- `Send_DI` — single-handler Send through injected `IMediator` (scope resolution per call). **Already partially exists**; rename `ZeroAllocMediator_Send_DI` and add a fair scope (call `provider.CreateScope()` per invocation to model real ASP.NET behavior).
- `SendPipeline_DI` — Send through `IMediator` with a pipeline behavior in the chain.
- `Publish_Single_DI` — single-handler Publish through `IMediator`.
- `Publish_Multi_DI` — multi-handler Publish through `IMediator`.
- `Stream_DI` — `CreateStream` through `IMediator`.
- `Setup_AssemblyScan` (one-time) — `IterationSetup` measuring cost of `services.AddMediator().RegisterHandlersFromAssembly(asm)` for a 20-handler assembly. Documents the startup cost (one-time, not on hot path) so users can size what they're paying.

**Acceptance criteria for the new DI path:**
- `Send_DI` ≤ 3× the static `Send` baseline (DI lookup overhead is acceptable; MediatR is currently ~30×).
- `Publish_*_DI` zero-allocation on the dispatch path (handler resolution allocates per MS DI; we measure that overhead but it should be ≤ 80 B / call for one handler).
- `Stream_DI` matches static `Stream` allocation (iterator dominates; DI lookup is one-time per stream).
- No regression on existing static-path benchmarks. The static path is unchanged; if existing numbers move, generator changes leaked.

Commit baseline numbers from `main` and post-change numbers in the PR description so reviewers can verify the DI path didn't regress static-path perf.

## Migration guide (CHANGELOG entry)

> ### 3.1.0 — DI integration
>
> **`IMediator` is now transient.** Inject it instead of the static `Mediator` class for ASP.NET / hosted apps — handlers resolve from the caller's scope (the request scope in ASP.NET; whatever scope you create with `IServiceScopeFactory` in a worker).
>
> **Assembly-scan registration:**
> ```csharp
> services.AddMediator()
>     .RegisterHandlersFromAssembly(typeof(Program).Assembly);
> ```
>
> **Drop the `null!` parameterless ctor.** If you only call your handler via injected `IMediator`, you can now delete the `internal MyHandler() : this(null!, null!) { }` shim. Build will tell you (`ZAM007`) if a static `Mediator.Send()` site needs a factory.
>
> **Static `Mediator.Send()` is unchanged** — keep using it where you want zero-alloc / no-DI dispatch.
